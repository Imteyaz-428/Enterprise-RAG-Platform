# Week 2 - Day 3: Vector Database Integration (pgvector)

---

## Q1. What is a Vector Database?

**Answer**

A Vector Database is a database designed to store, index, and search vector embeddings efficiently.

Instead of storing only text or numbers, it stores high-dimensional vectors generated by embedding models.

These vectors are used to perform semantic similarity search.

---

## Q2. Why do we need a Vector Database in a RAG system?

**Answer**

After generating embeddings, we need a place to store them and efficiently retrieve the most relevant ones.

A vector database allows us to:

- Store embeddings.
- Perform similarity search.
- Retrieve the most relevant chunks for a user query.

Without a vector database, semantic retrieval would not be possible.

---

## Q3. What is pgvector?

**Answer**

pgvector is a PostgreSQL extension that adds support for vector data types and vector similarity search.

It allows PostgreSQL to store embeddings and perform semantic search without requiring a separate vector database.

Example:

```python
embedding = Column(Vector(1024))
```

---

## Q4. Why did you choose pgvector instead of Pinecone or Milvus?

**Answer**

My project already stores:

- Organizations
- Users
- Documents
- Document Metadata

inside PostgreSQL.

Using pgvector allows me to keep relational data and embeddings inside the same database.

Advantages:

- Simpler architecture
- Easier maintenance
- No synchronization between two databases
- Lower infrastructure complexity

For enterprise applications with moderate data sizes, pgvector is an excellent choice.

---

### Follow-up Question

**When would you choose Pinecone, Milvus, or Weaviate instead?**

**Answer**

Dedicated vector databases are more suitable when:

- Billions of vectors must be stored.
- Extremely low-latency retrieval is required.
- Distributed indexing is needed.
- Horizontal scaling is important.

For small and medium enterprise applications, pgvector is usually sufficient.

## Q5. Why did you use Docker for PostgreSQL?

**Answer**

Docker provides an isolated PostgreSQL environment with pgvector already installed.

Benefits:

- Consistent environment.
- Easy setup.
- No manual PostgreSQL configuration.
- Easy project portability.

## Q6. Explain your DocumentChunk model.

**Answer**

Each document is divided into multiple chunks.

Each chunk stores:

- Document ID
- Chunk Index
- Chunk Text
- Embedding Vector

Example:

```text
Document

↓

Chunk 1
Embedding

Chunk 2
Embedding

Chunk 3
Embedding
```

This structure allows efficient retrieval of only the relevant parts of a document.

---

## Q7. Why store both Chunk Text and Embedding?

**Answer**

The embedding is used only for similarity search.

After retrieval, the LLM needs the original text to generate the final answer.

Therefore we store:

- Chunk Text → Sent to the LLM.
- Embedding → Used for similarity search.

## Q8. Explain the complete Document Ingestion Pipeline.

**Answer**

```text
Upload PDF

↓

Validate PDF

↓

Store File

↓

Extract Text

↓

Chunk Text

↓

Generate Embeddings

↓

Store Chunks & Embeddings in PostgreSQL
```

This prepares documents for semantic retrieval.

---

## Q9. Why did you switch from single embedding generation to batch embedding generation?

**Answer**

Initially, I generated one embedding per API call to understand the embedding process.

Later, I switched to batch embedding generation because it:

- Reduces API calls.
- Improves performance.
- Reduces latency.
- Is the recommended production approach.
